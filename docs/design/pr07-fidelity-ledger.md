## 关键区域对齐

| 区域 | 概念目标 | React 实现 |
| --- | --- | --- |
| 顶部上下文 | 工程、阶段状态、执行器和主要动作同时可见 | 读取真实 `ProjectDescriptor`、`WorkflowStageState` 和所选 `StageRun.executor`；演示模式有明确标记 |
| 阶段导航 | 九阶段及采用、待审核、过期、失败状态可扫描 | 沿用类型化 workflow snapshot；动作执行和数据加载期间禁用切换 |
| 中央工作区 | Input / Config / Output / Preview / History / Compare / Review 七个工作区 | 七个页签全部可操作；版本 A/B 均来自不可变 `StageRun` 历史 |
| 版本比较 | 候选版本与已采用版本并排对比 | 比较配置修订、变化键、输入、产物、警告和可用的演示预览 |
| 运行历史 | 历史版本不可变，采用版本明确 | 右侧历史面板显示 run 状态、配置快照、执行器、产物和最新审核 |
| 审核 | 采用、请求修改、拒绝均写入审核记录 | 采用必须显式选择至少一个产物；每次提交复用稳定 `reviewId` 处理响应丢失 |
| 重生成 | 执行前展示影响范围，确认后创建新任务 | 先调用 `preview_regeneration`，确认后以新 `runId + idempotencyKey` 调用任务队列；不覆盖历史 |
| 活动区 | 任务、日志、产物保持可观察 | 沿用 PR06 的 Job/Events/Artifact 活动区和取消、恢复入口 |

## 有意差异

1. 概念稿用双纸张正文强调视觉方向；真实桌面路径当前只能通过 Artifact Store 取得元数据、`contentUri` 和 `contentAvailable`，还没有有界内容字节读取命令。因此 React 实现只在明确的演示模式展示示例正文，真实工程显示元数据与能力边界，不伪造内容。
2. 右侧历史面板在宽度小于 1180px 时隐藏；同样的历史、比较、审核和重生成操作仍可从中央页签完成，避免窄屏出现三栏挤压。
3. 配置编辑采用 JSON 对象编辑器，因为当前阶段契约只提供 `configSchemaRef`，尚未提供可直接渲染的字段级 UI Schema。保存仍使用乐观修订与 `DecisionRecord`，没有绕过契约。

## 浏览器交互验收

| 场景 | 结果 |
| --- | --- |
| 打开月球城市演示工程 | 默认定位 `script` 待审核阶段，加载 3 个不可变运行和 2 个候选产物 |
| 版本比较 | `run_script_004` 与已采用 `run_script_003` 显示 2 个配置键变化及各自快照 |
| 配置缺少变更理由 | 阻止保存并显示可访问错误提示，不创建新修订 |
| 请求修改 | 写入不可变审核记录，既有采用版本保持不变 |
| 重生成影响预览 | 展示当前阶段与所有下游受影响阶段，预览本身不创建任务 |
| 确认重生成 | 创建新的 queued Job；活动区变为 1 个运行中，历史仍为原有 3 个终态运行 |
| 停止新任务 | Job 变为 canceled，运行预留和既有历史仍保留 |
| 1180 / 900 / 680px 断点 | 中央七页签、主版本选择和九阶段导航仍可访问；右侧摘要按设计收起 |
| 浏览器控制台 | 最终新页面检查无 error / warning |

最终控制台结论必须以完成代码后的新页面为准；开发期间的旧页面会保留 React HMR 改变 Hook 结构时产生的历史日志，不作为最终验收结果。
